The List Provinces endpoint returns the full catalogue of geographic province entries stored in Ship Quote. Each province object includes the two-digit postal prefix, the normalized name used for internal search, and the `adminFullCode` ŌĆö a compound code like `ES-GR` or `ES-M` that you pass directly to `POST /rates/compareByProvinceCode` when you already know the destination province. The Spanish province dataset (all 52 provinces including the Balearic and Canary Islands) is seeded automatically on first startup and is served from an in-memory cache, so this endpoint returns results with no database query.

## Endpoint

```
GET /api/v1/locations/provinces
```

## Query Parameters

<ParamField query="countryCode" type="string">
  Filter results to provinces belonging to a specific country. Pass the
  ISO-3166-1 alpha-2 code in uppercase (e.g. `ES`). When omitted, all provinces
  across all countries are returned.
</ParamField>

<Note>
  The `countryCode` filter is applied client-side against the in-memory province
  list. The current seed dataset contains Spanish provinces only (`countryCode:
  "ES"`). Provinces for other countries can be added via
  [`POST /api/v1/locations`](/api/locations/create).
</Note>

## Response

Returns `HTTP 200 OK` with a JSON array of province objects.

<ResponseField name="_id" type="string">
  MongoDB ObjectId of the province document.
</ResponseField>

<ResponseField name="countryCode" type="string">
  Uppercased ISO-2 country code (e.g. `ES`).
</ResponseField>

<ResponseField name="countryName" type="string">
  Country name as stored (e.g. `Spain`).
</ResponseField>

<ResponseField name="adminCode" type="string">
  Short administrative region code (e.g. `VI` for ├ülava, `AB` for Albacete,
  `M` for Madrid). Used together with `countryCode` to build `adminFullCode`.
</ResponseField>

<ResponseField name="adminFullCode" type="string">
  Fully-qualified province code with country prefix (e.g. `ES-VI`, `ES-AB`,
  `ES-M`). **This is the value you pass as `province` when calling
  `POST /rates/compareByProvinceCode`.**
</ResponseField>

<ResponseField name="name" type="string">
  Display name of the province in Title Case (e.g. `├ülava`, `Albacete`).
</ResponseField>

<ResponseField name="normalizedName" type="string">
  Lowercase, accent-stripped, underscore-separated version of the name used
  for internal indexing and search (e.g. `alava`, `albacete`).
</ResponseField>

<ResponseField name="postalCode" type="string">
  Two-digit postal prefix that identifies the province. For example, any
  5-digit postal code beginning with `01` belongs to ├ülava, and any code
  beginning with `28` belongs to Madrid. Pass this prefix to
  [`GET /api/v1/locations/provincesByPostalCode/:postalCode`](/api/locations/province-by-postal-code)
  for reverse lookup.
</ResponseField>

<ResponseField name="type" type="string">
  Geographic classification. For the seeded Spanish dataset this is always
  `province`. Custom entries may be `state` or `region`.
</ResponseField>

<ResponseField name="createdAt" type="string">
  ISO 8601 timestamp of when the document was created.
</ResponseField>

<ResponseField name="updatedAt" type="string">
  ISO 8601 timestamp of the most recent update.
</ResponseField>

<Tip>
  Use the `adminFullCode` values from this response (e.g. `ES-GR`, `ES-M`) as
  the `province` field when calling `POST /rates/compareByProvinceCode`. This
  workflow is recommended when you already know the destination province ŌĆö it
  skips the postal-code resolution step and gives you direct control over which
  province's rate table is applied.
</Tip>

<Info>
  Province records are loaded into memory from MongoDB at server boot via
  `loadProvinces()` and stored in a `Map` keyed by two-digit postal prefix.
  The `getProvinces()` call behind this endpoint returns the full in-memory
  array ŌĆö no additional database query is issued at request time.
</Info>
